# 1/6/12/24時間後 降水確率・気温予測 LightGBM 学習パイプライン設計

## 背景・目的

現状のダッシュボード（雨ページ・熱中症ページ）は観測値の可視化のみ。1/6/12/24時間後の
**降水確率**と**気温**を予測するモデルを学習し、将来ダッシュボードへ予測表示する土台を作る。

今回のスコープは**学習パイプラインのみ**。`server.py` への推論組み込みは別タスク。

2015〜2025年の11年分の観測データ（`observations.csv` 官署 / `observations_amedas.csv`
アメダス）が揃ったため、これを使って学習する。

## モデル設計

### 方針: LightGBM（勾配ブースティング木）

表形式・地点特徴量の予測ではGBDTがMLPをよく上回り、CPUで高速、欠測をそのまま扱える。
PyTorchのMLPは採用しない。

### 対象と本数

予測対象 × 予測時間ごとに独立したモデルを1本ずつ学習する（LightGBMはマルチ出力
非対応のため）。

- 降水: `precip >= 0.1` の二値分類 × [1h, 6h, 12h, 24h] = **4本**
- 気温: 回帰 × [1h, 6h, 12h, 24h] = **4本**
- 計 **8本**

官署・アメダス共通の単一モデル。地点差は静的特徴量（緯度経度・標高）で吸収させる。

### 特徴量

官署・アメダス両方が持つ項目のみを使う（湿度・日射はアメダスに無いため不使用、
補完処理は行わない）。

- 時系列特徴量: 基準時刻から直近24時間分の `temp / precip / wind_speed`（3種 × 24時間 = 72次元）
  - 欠測は NaN のまま LightGBM に渡す（GBDTは欠測をネイティブに扱う）
- 静的特徴量: `lat, lon, elev`（`stations*.csv` から地点IDで結合、3次元）
- 時刻特徴量: `month`（1-12）, `hour`（0-23）を整数のまま categorical 指定（4次元ではなく2列）

合計 約77列。

### 正解ラベル

- 降水確率の正解: 対象時刻の `precip >= 0.1`（既存 `isRaining` と同閾値）を 1/0
- 気温の正解: 対象時刻の `temp` 実測値

### 損失

LightGBM 標準の目的関数を使う。降水分類は `binary`（logloss）、気温回帰は
`regression`（L2）。

## データパイプライン

### 読み込み

`observations.csv` / `observations_amedas.csv` から**必要列のみ**読む
（`datetime, station_id, temp, precip, wind_speed`）。文字列列（`name, wind_dir` 等）は
捨てるためメモリに乗る。特徴量行列は float32 で保持する（全サンプルで約3.2GB）。

CSVは地点×年でまとまっており全体ソートされていない。train/test分割は各行の
`datetime` を見て振り分ける（ファイル位置に依存しない）。

### サンプル抽出条件

地点×基準時刻の組ごとに、以下を満たす場合のみ学習サンプルとする:

- 基準時刻から 1/6/12/24 時間後の観測（`precip`, `temp`）が**すべて存在**する
- 直近24時間の履歴は欠測があっても可（NaNのまま特徴量に入れる）

### train/test 分割

日付ベースで固定分割する（ランダムシャッフルは同日前後リークのため使わない）。

- **train: 2015-01-01 〜 2024-12-31（10年）**
- **test(検証): 2025-01-01 〜 2025-12-31（直近1年）**
- 2026年の端数（官署の一部のみ）は除外

### サンプル数の制御

デフォルトは**全サンプルを使う**（間引きの仕組みは作らない）。保険として
`max_samples` パラメータを1つ用意し、指定時は train からランダムに上限件数まで
絞る。用途は (1) 自己チェック/開発時に小さく速く回す、(2) 万一メモリ不足のときの
頭打ち。

## ベースライン & 評価

### persistence ベースライン（必須）

学習前に、素朴な persistence 予測を test で評価して基準値を出す:

- 気温: 「t+k の気温 = t の気温」
- 降水: 「t+k の降水確率 = t の降水状態（0/1）」

### 評価指標

- 気温（回帰）: **MAE / RMSE**
- 降水（分類）: **AUC / Brier score**

### 比較

学習後、**8本すべてで LightGBM と persistence を並べて表示**する。LightGBM が
persistence を超えていない対象があれば警告として明示する（超えられない予測は
ダッシュボードに出す価値がないため）。

## ファイル構成

```
backend/ml/
  dataset.py    # CSV読込 → 24時間窓化 → サンプル行列 → 日付分割 → max_samples適用
  baseline.py   # persistence 予測の指標計算
  train.py      # 8本学習 → test評価 → baseline比較 → models/ と metrics.json 保存
  models/       # lgb_precip_1h.txt ... lgb_temp_24h.txt（8ファイル）
  metrics.json  # 8本 × (LightGBM, persistence) の指標
```

`requirements.txt` に `lightgbm, pandas, numpy, scikit-learn` を追加
（AUC / MAE / Brier は scikit-learn を使用）。

## 自己チェック

`train.py` の `if __name__ == "__main__"` または `backend/ml/test_train.py` に、
小さい `max_samples` でパイプライン全体を実行し以下を `assert` する最小テストを置く:

- 学習が例外なく完走すること
- test の各指標が有限値（NaN/Inf でない）であること
- 気温の MAE が persistence の MAE 以下であること（学習が機能している最低限の確認。
  予測精度そのものの保証はしない）

フレームワーク・フィクスチャは使わない。
