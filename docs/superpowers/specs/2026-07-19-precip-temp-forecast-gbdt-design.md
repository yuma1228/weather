# 1/6/12/24時間後 降水確率・気温予測 LightGBM 学習パイプライン設計

## 背景・目的

現状のダッシュボード（雨ページ・熱中症ページ）は観測値の可視化のみ。1/6/12/24時間後の
**降水確率**と**気温**を予測するモデルを学習し、将来ダッシュボードへ予測表示する土台を作る。

今回のスコープは**学習パイプラインのみ**。`server.py` への推論組み込みは別タスク。

2015〜2025年の11年分のアメダス観測データ（`observations_amedas.csv`）を使って学習する。
官署（`observations.csv`）は使わない: 予測に使う特徴量は temp/precip/wind のみで、
官署だけが持つ湿度・日射は不使用のため、官署を足しても新しい特徴量は増えず
サンプル数が約1割増えるだけ。2ファイル読み込みの手間に見合わないため除外する。
アメダス約1,300地点で全国を密にカバーしており、これで十分。

## モデル設計

### 方針: LightGBM（勾配ブースティング木）

表形式・地点特徴量の予測ではGBDTがMLPをよく上回り、CPUで高速、欠測をそのまま扱える。
PyTorchのMLPは採用しない。

### 対象と本数

予測対象 × 予測時間ごとに独立したモデルを1本ずつ学習する（LightGBMはマルチ出力
非対応のため）。

- 降水: `precip >= 0.1` の二値分類 × [1h, 6h, 12h, 24h] = **4本**
- 気温: 回帰 × [1h, 6h, 12h, 24h] = **4本**
- 計 **8本**

アメダス全地点を1つの単一モデルで学習する。地点差は静的特徴量（緯度経度・標高）で
吸収させる。

### 特徴量

アメダスが持つ項目のみを使う（湿度・日射は無いため不使用、補完処理は行わない）。
欠測はすべて NaN のまま LightGBM に渡す（GBDTは欠測をネイティブに扱う）。

特徴量は**2段階**で組む。第1段階（単独地点）でパイプラインを通して persistence
ベースラインを超えることを確認し、第2段階（近傍地点）でそれを積み増す。効果は
「baseline比」「近傍追加でどれだけ改善したか」として測る。

**第1段階（単独地点、約77列）:**

- 時系列特徴量: 基準時刻から直近24時間分の対象地点の `temp / precip / wind_speed`
  （3種 × 24時間 = 72次元）
- 静的特徴量: `lat, lon, elev`（`stations_amedas.csv` から地点IDで結合、3次元）
- 時刻特徴量: `month`（1-12）, `hour`（0-23）を整数のまま categorical 指定（2列）

**第2段階（近傍K地点、+ K×3列）:**

- 対象地点の**最寄りK地点**（デフォルト K=5）の、基準時刻の `temp / precip / wind_speed`
  （K × 3 = 15列、デフォルト時）
- 近傍は `stations_amedas.csv` の緯度経度から一度だけ全地点ペアの haversine 距離を
  計算し、各地点について近い順に K 個の `station_id` を確定した表
  `{station_id: [近傍1, ..., 近傍K]}` を作って引く（観測データ不要、前計算）
- 列順は常に「1番近い→K番目に近い」で固定（対象地点が変わっても列 j = j番目に
  近い地点、の意味を一定に保つ）
- 近傍地点が基準時刻に欠測なら NaN のまま
- 雨は上流→下流へ移動するため、近傍の現在降水は降水予測に特に効く見込み

第1段階=約77列、第2段階込み=約92列（K=5時）。

### 正解ラベル

- 降水確率の正解: 対象時刻の `precip >= 0.1`（既存 `isRaining` と同閾値）を 1/0
- 気温の正解: 対象時刻の `temp` 実測値

### 損失

LightGBM 標準の目的関数を使う。降水分類は `binary`（logloss）、気温回帰は
`regression`（L2）。

## データパイプライン

### 読み込み

`observations_amedas.csv` から**必要列のみ**読む
（`datetime, station_id, temp, precip, wind_speed`）。文字列列（`name, wind_dir` 等）は
捨てるためメモリに乗る。静的特徴量は `stations_amedas.csv` を地点IDで結合。

CSVヘッダにBOMが付いている（`﻿datetime`）ため、読み込みは `encoding="utf-8-sig"` を
指定する（指定しないと先頭列名が `﻿datetime` になり `usecols` / 列選択が壊れる）。

**フルアーカイブは約1.1億行（109,064,593行）ある。** 全サンプルの特徴量行列を
materialize すると float32 で約34GB になり CPU メモリに乗らないため、後述のとおり
**必ずサンプリングする**。

CSVは地点×年でまとまっており全体ソートされていない。train/test分割は各行の
`datetime` を見て振り分ける（ファイル位置に依存しない）。

### サンプル抽出条件

地点×基準時刻の組ごとに、以下を満たす場合のみ学習サンプルとする:

- 基準時刻から 1/6/12/24 時間後の観測（`precip`, `temp`）が**すべて存在**する
- 直近24時間の履歴は欠測があっても可（NaNのまま特徴量に入れる）

### train/test 分割

日付ベースで固定分割する（ランダムシャッフルは同日前後リークのため使わない）。

- **train: 2015-01-01 〜 2024-12-31（10年）**
- **test(検証): 2025-01-01 〜 2025-12-31（直近1年）**
- アメダスは2015〜2025でほぼ揃う。2026年の端数行があれば除外

### サンプリング（必須）

1.1億行を全部は使えず、また GBDT は特徴量77列程度なら数百万サンプルで指標が
ほぼ頭打ちになるため、**ランダムに一様サンプリングする**。

**サンプリングは窓化の前に行う。** 順序は「有効な基準時刻を列挙（下記サンプル抽出
条件を満たす地点×時刻）→ そこから一様サンプリング → 選ばれた点だけ 24時間窓化・
近傍結合して特徴量行列を作る」。先に全行を窓化すると 1.1億×72列を materialize して
しまい、避けたい34GBを踏む。有効基準時刻の列挙は地点ごとの時系列を1回ずつ走査すれば
足りる（全窓を保持する必要はない）。

- **train**: 2015〜2024 から一様ランダムに `n_train` 件（デフォルト **500万**、
  500万×約92列×float32 ≈ 1.7GB）
- **test**: 2025 から一様ランダムに `n_test` 件（デフォルト **100万**。指標は
  この規模で安定する）
- `n_train` / `n_test` はパラメータ。小さくすれば自己チェックが一瞬で終わる

**層化・オーバーサンプリングはしない。** 降水の正例（雨）だけを増やすと出現率が
人工的に上がり、出力確率が雨寄りに歪んで校正が崩れる（Brier が悪化する）。一様
サンプリングなら 500万件でも正例は数十万件あり十分。レアな大雨で不足が出た場合の
み、リサンプルではなく `scale_pos_weight` での重み付けを将来検討する（スコープ外）。

### サンプル数の妥当性確認（learning curve）

「何件で足りるか」は勘で決めず実測する。`n_train` を 100万 / 300万 / 500万 と
変えて学習し、test 指標を並べる補助を用意する。**まだ改善していれば増やす、平らなら
足りている**証拠とする。

## ベースライン & 評価

### persistence ベースライン（必須）

学習前に、素朴な persistence 予測を test で評価して基準値を出す:

- 気温: 「t+k の気温 = t の気温」
- 降水: 「t+k の降水確率 = t の降水状態（0/1）」

### 評価指標

- 気温（回帰）: **MAE / RMSE**
- 降水（分類）: **AUC / Brier score**

### 比較

学習後、**8本すべてで LightGBM と persistence を並べて表示**する。LightGBM が
persistence を超えていない対象があれば警告として明示する（超えられない予測は
ダッシュボードに出す価値がないため）。

## ファイル構成

```
backend/ml/
  neighbors.py  # stations_amedas.csv → 全地点の最寄りK地点表（haversine、前計算）
  dataset.py    # CSV読込 → 日付分割 → 有効基準時刻の列挙 → サンプリング → (選択点のみ)24時間窓化・近傍結合 → サンプル行列
  baseline.py   # persistence 予測の指標計算
  train.py      # 8本学習 → test評価 → baseline比較 → models/ と metrics.json 保存
  models/       # lgb_precip_1h.txt ... lgb_temp_24h.txt（8ファイル）
  metrics.json  # 8本 × (LightGBM, persistence) の指標
```

`requirements.txt` に `lightgbm, pandas, numpy, scikit-learn` を追加
（AUC / MAE / Brier は scikit-learn を使用）。

## 自己チェック

`train.py` の `if __name__ == "__main__"` または `backend/ml/test_train.py` に、
小さい `n_train` / `n_test` でパイプライン全体を実行し以下を `assert` する最小テストを置く:

- 学習が例外なく完走すること
- test の各指標が有限値（NaN/Inf でない）であること
- **12h先**気温の MAE が persistence の MAE 以下であること（学習が機能している
  最低限の確認。予測精度そのものの保証はしない）。1h/24h先は使わない: 気温は日周期を
  持つため persistence が 1h先（ほぼ不変）と 24h先（同時刻で日周期が一致）で強く、小さい
  `n_train` の自己チェックでは正常なパイプラインでも超えられず assert が誤爆する。12h先は
  正午↔深夜が逆相で persistence が最も弱く、確実に上回る

フレームワーク・フィクスチャは使わない。
